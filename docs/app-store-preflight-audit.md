# Lullaby — App Store Submission Preflight Audit

> Pre-submission audit produced with the `app-store-preflight-skills` checklists.
> This is an assessment document only — it changes no code. Every finding ties to
> an Apple App Review Guideline number and a skill rule.

## Context

**App profile (from the repo):**
- **App:** Lullaby — logs Feed / Sleep / Diaper / Pump for a baby, plus an AI-phrased "Reassure night read." Bundle id `com.bizhanash.lullaby` (`app.json:12`, authoritative). Version `1.0.0`.
- **Stack:** Expo SDK 56 / RN 0.85, **managed / CNG** (`app.json` is the source of truth; `ios/` is git-ignored and regenerated by EAS). iPhone-only, portrait, min iOS ~16.4.
- **Backend:** Supabase (auth + Postgres + RLS + edge functions). AI = **Anthropic Claude** (`claude-haiku-4-5`) via a **server-side** Supabase edge function; key never on client.
- **Monetization:** **RevenueCat** subscriptions (`react-native-purchases` 10.4.0).
- **Auth:** email/password, **Sign in with Apple**, Google OAuth (env-gated), local/guest mode.

**Submission posture assumed (confirmed with the team — drives severities below):**
- **Pro subscriptions: LIVE at launch** → full Guideline 3.1.2 subscription surface is in scope.
- **AI Reassure: ENABLED (calls Claude)** → full AI (Guideline 5 / 1.4.1) + health surface is in scope.
- **China mainland: EXCLUDED** → the Chinese DST/MIIT risk is removed *provided China is actually deselected in App Store Connect* (action item, not a blocker).

---

## Verdict

**NOT READY.** Given the posture (subs live + AI on), there are **3 blockers**, each a near-certain rejection: the in-app Privacy Policy / Terms / support links resolve to a **for-sale parked domain** (`lullaby.app` → GoDaddy for-sale, verified live); the **AI system prompts + clinical knowledge base are explicitly placeholders "pending clinician review"** while the health-adjacent AI ships enabled; and the **subscription products aren't provably live/configured** in the reviewed build (with a paywall still captioned *"Coming later."*). The good news: the engineering is genuinely review-aware — SIWA, real account deletion, minimal permissions, server-side AI key, strong disclaimers, RLS, no external-payment path are all already correct. The blockers are mostly **go-live configuration + content sign-off**, not rearchitecting.

## Findings (BLOCKER → LOW)

### 1 — BLOCKER · Legal & support links resolve to a for-sale domain
**Guideline 3.1.2 (subscriptions need functional ToS + PP links) + 5.1.1(i) (privacy policy in-app) + 1.5 (support contact).** Skill rules: `subscription/missing_tos_pp.md`, `privacy/*`.
- **What's wrong:** `src/lib/appLinks.ts:21-23` defaults Privacy Policy → `https://lullaby.app/privacy`, Terms → `https://lullaby.app/terms`, support → `support@lullaby.app`. **Verified live:** all of `lullaby.app` `307`-redirects to `forsale.godaddy.com/forsale/lullaby.app` — the domain is **parked and for sale; the developer does not own it.** So both legal links are dead, and the support mailbox will bounce. These render on the **paywall** (`PaywallSheet.tsx:22,343,359`) and in **Settings** (`settings.tsx:480,482,495`). With subscriptions live, dead ToS/PP links on the purchase screen are an automatic 3.1.2 rejection; a dead privacy policy is a 5.1.1 rejection.
- **Env caveat:** the defaults are overridable via `EXPO_PUBLIC_PRIVACY_POLICY_URL` / `EXPO_PUBLIC_TERMS_URL` / `EXPO_PUBLIC_SUPPORT_EMAIL`. `.env` is read-denied, so the *actual* shipped values are unverified — but as written, and per `appLinks.ts:8-12` + `docs/release-env.md`, no real pages exist yet.
- **Fix:** register a domain you own → host a real **Privacy Policy** (must disclose the health-adjacent data collected *and* that night tallies go to **Anthropic/Claude** — see #5) and **Terms of Use / EULA** → set the three `EXPO_PUBLIC_*` vars for the production build → set the **Privacy Policy URL** in App Store Connect → add ToS + PP links to the **App Store description** → verify each URL resolves to real content. **Effort: M.**

### 2 — BLOCKER · AI ships enabled on placeholder clinical prompts
**Guideline 1.4.1 (health content must be credible; remind users to consult a professional) + 2.1 (no placeholder content) + 2.3.1 (disclose all AI in review notes).** Skill: `by-app-type/ai_apps.md`, `by-app-type/health_fitness.md`.
- **What's wrong:** with AI enabled, the Reassure night read calls Claude using system prompts + a clinical knowledge base explicitly stamped `PLACEHOLDER, PENDING CLINICIAN REVIEW` (`src/features/reassure/content/kb.ts:4`; `reassureContent.ts:13-15,151-155`; `reassure.tsx:13`). The output is health-adjacent guidance to parents about an infant. Shipping unreviewed clinical/AI content in a health-adjacent app is exactly the 1.4.1/2.1 pattern.
- **Strong mitigations already present (keep them):** explicit user consent gate, server kill-switch, red-flag substring pre-scan (fever / "not breathing" / seizure / blood…), a system prompt that forbids diagnosis + judgement words, an output validator (length cap + banned-vocab + new-medical-claim reject → falls back to local copy), a 90-day audit log, and consistent *"General information, not medical advice — never a diagnosis"* disclaimers (`AiConsentCard.tsx:74`, `AiReadNote.tsx:28`). The **architecture is safe; the source content is unsigned-off.**
- **Fix:** get the system prompts + KB **clinician-reviewed** and flip status out of draft/placeholder before enabling; keep the disclaimers and the "call your doctor / pediatrician" reminders; in review notes, **explicitly document the AI feature and Anthropic as the provider** (hidden AI is its own rejection). **Effort: L (external clinician dependency).**

### 3 — BLOCKER · Live subscription products not provably configured for review
**Guideline 3.1.2 + 2.1(b) (all IAP items must be complete, visible, functional to the reviewer).** Skill: `by-app-type/subscription_iap.md`.
- **What's wrong:** the RevenueCat integration is real and correct in code (`src/lib/revenueCat.ts`), but **no products exist in the repo** (expected — they live in the RevenueCat + ASC dashboards). The paywall only shows real packages when the build carries real RC keys **and** the offering `default` / entitlement `pro` (`proConfig.ts` defaults) has live products. If keys/products are missing, the sheet falls to `"Subscriptions are not configured in this build yet."` (`PaywallSheet.tsx:38`) or `"…aren't available right now."` — a dead-end upsell the repo's own `docs/release-env.md` says to **"never ship on purpose."** A reviewer who taps upgrade and hits that = 2.1 rejection.
- **Fix:** create the subscription products in App Store Connect, attach them to the RevenueCat `default` offering / `pro` entitlement, get them to "Ready to Submit," **build production with real `EXPO_PUBLIC_REVENUECAT_IOS_API_KEY` + `EXPO_PUBLIC_PRO_ENABLED=1`**, and confirm the paywall shows real store price strings (never the unconfigured/unavailable state). Provide the reviewer a path + demo account that reaches a sandbox purchase. **Effort: M.**

### 4 — HIGH · Live paywall says "Coming later." / "Soon"
**Guideline 2.3.2 (accurate feature availability) + 3.1.2.**
- `PaywallSheet.tsx:29` subtitle ends *"…export-ready summaries. **Coming later.**"* and a **"Soon"** badge renders (`:225-239`) on a screen that sells a real subscription. Contradictory/misleading about what the purchase delivers.
- **Fix:** remove "Coming later." + the "Soon" badge for the live paywall; ensure the three benefits (fuller history, weekly recaps, export summaries) are actually delivered by the `pro` entitlement gates in `src/lib/proGates.ts`. **Effort: S.**

### 5 — HIGH · App Privacy label must match actual collection incl. third-party AI
**Guideline 5.1.1.** Skill: `privacy/unnecessary_data.md`, `by-app-type/ai_apps.md`.
- The app collects **email** (auth), **baby name + birth date**, **feed/sleep/diaper/pump logs** (health-adjacent User Content), **identifiers**, and coarse **analytics**; with AI on it sends night **tallies + age band to Anthropic** (a third party). The ASC App Privacy questionnaire must reflect all of this, including data shared with a third party for the AI feature. `app.json` `privacyManifests` declares `NSPrivacyCollectedDataTypes: []` (understates collection) — the ASC label is what Apple checks, so make it accurate there. Cannot verify ASC (no access) → **must-do + verify.** **Effort: M.**

### 6 — HIGH · New-submission review notes incomplete
**Guideline 2.1 / 2.3.1.** Skill: `metadata/review_notes_new_submission.md`.
- A first submission needs all six sections. This app triggers most of them: account create/**delete**, a **subscription** purchase, **sensitive-capability** prompts (mic/speech), and **AI**. Provide: (1) device screen recording of the core flow; (2) purpose; (3) **demo credentials** that reach the paywall, the AI night read, and account deletion; (4) external services = **Supabase, RevenueCat, Anthropic/Claude**; (5) regional note; (6) health-adjacent note. **Effort: M.**

### 7 — MEDIUM · No consent / legal links at account creation
**Guideline 5.1.1(ii).**
- Privacy/Terms links appear only in Settings (`settings.tsx:480,482`); the sign-up surfaces (`AccountEntryScreen.tsx`, `AuthScreen.tsx`) have only a decorative "Privacy-first" chip — no link, no "by continuing you agree…" copy. Given email + health-adjacent data at signup, add a consent line + PP/Terms links to the account-entry screen. **Effort: S.**

### 8 — MEDIUM · China must actually be deselected (DST)
**Guideline 5 (Chinese deep-synthesis regulations).** Skill: `metadata/china_storefront.md`.
- Decision is to exclude China. This is only safe if China mainland is **actually deselected** in ASC → Pricing & Availability; otherwise the enabled Claude AI is unlicensed DST in China = rejection. Confirm it, and keep AI-provider brand names out of metadata as a hygiene measure. **Effort: S (a checkbox — but forgetting it re-opens a blocker).**

### 9 — MEDIUM · Confirm mic/speech + AI consent surfacing
**Guideline 2.5.14 (consent to record activity) + ai 2.5.14.**
- Usage strings are present and specific (`NSMicrophoneUsageDescription`, `NSSpeechRecognitionUsageDescription`, `app.json:16-17`), and there's an AI consent card. Apple Speech may send audio to Apple's servers — verify the voice-ask path shows clear consent before recording, and document mic/speech + AI processing in review notes so the permission prompt isn't a surprise to the reviewer. **Effort: S.**

### 10 — LOW · Clean prebuild; verify archive contents
**Guideline 2.4.5 / 5.1.1 (privacy manifest).** Skill: `entitlements/unused_entitlements.md`, `privacy/privacy_manifest.md`.
- The on-disk `ios/` prebuild is stale: bundle id `com.lullaby.app` vs `app.json` `com.bizhanash.lullaby`; deployment target 16.4 vs `LSMinimumSystemVersion` 12.0; `MARKETING_VERSION` 1.0 vs 1.0.0; `ITSAppUsesNonExemptEncryption:false` and the `privacyManifests` block are declared in `app.json` but **not yet materialized** in the stale prebuild. CNG regenerates on EAS build, so `app.json` wins — but run a fresh build and **verify** the archive carries the right bundle id (matching the ASC record), a generated `PrivacyInfo.xcprivacy` (UserDefaults `CA92.1` ✓), and `ITSAppUsesNonExemptEncryption=false`. The only entitlement is `com.apple.developer.applesignin` — used (SIWA), so no unused-entitlement risk. **Effort: S.**

### 11 — LOW · Debug logging leftovers
**Guideline 2.1.** ~29 `console.*` in `src/` (many dev-gated). Cosmetic; confirm none log tokens/PII in a release build. **Effort: S.**

## What's Already Good (won't trigger rejection)

- **4.8 Sign in with Apple** is present alongside Google OAuth (`AuthProvider.tsx:620-667`), with the native `AppleAuthenticationButton`; **no post-SIWA name/email re-ask form** was found → also clean on **4.0**.
- **5.1.1(v) account deletion** is real: in-app two-step confirm → `supabase.rpc('delete_account')`, a `SECURITY DEFINER`, `auth.uid()`-scoped cascade delete (migration `20260703060000_delete_account.sql`), with an email fallback only on server failure.
- **Permissions are minimal** (mic + speech only), each with a specific, meaningful usage string; no camera/photos/location/contacts/notifications requested.
- **3.1.1 / 3.1.2 payment hygiene:** core logging is never gated (only depth/export), **Restore Purchases is always reachable** (`ProProvider.tsx:232-266`), auto-renew disclosure is present and subordinate to the store price, and there is **no external-payment / "contact us to upgrade" path** anywhere.
- **AI security:** API key is **server-side only** (`ANTHROPIC_API_KEY` via `Deno.env.get`), no PII/raw text sent to the model, layered guardrails + output validation + audit log, and clear "not medical advice / never a diagnosis" disclaimers.
- **Privacy manifest** declared, **`ITSAppUsesNonExemptEncryption:false`**, **no ATT/tracking SDKs** (no ATT prompt needed), `NSPrivacyTracking:false`.
- **No client-exposed secrets** (Supabase publishable anon key only), **RLS on every table**, pediatrician phone stored **locally only**.
- **Not Kids category**; audience is parents; **no data collected directly from children**.
- Verification tooling present: `npm run typecheck` (`tsc --noEmit`), `npm run lint` (`expo lint`), plus a custom smoke `check:local-interactions`. (No `test` script — documented gap.)

- **`.env` is read-denied:** the *actual* production values of `EXPO_PUBLIC_*` (legal URLs, RevenueCat keys, `PRO_ENABLED`, Google client id, Supabase keys) are unverified — findings #1/#3 hinge on these. Confirm the production build's env.
- **Legal adequacy** of the hosted PP/Terms (health data about infants + parents) — recommend real legal review, not just placeholder pages.
- **Clinician sign-off** on Reassure content is an external dependency (#2).
- **Real-device pass required:** launch, no-crash core flow, live paywall/sandbox purchase, SIWA relay acceptance, account deletion, mic/speech + AI consent prompts.
- **Developer identity (5.1.1(ix)):** health-adjacent apps are best submitted as a **legal entity**, not an individual — confirm the account type.
- **Git-history secret hygiene:** an older team baseline flags possible tracked secrets in commit `3c71f2d` (signing/maps creds) — verify nothing live remains in history; rotate if so.
